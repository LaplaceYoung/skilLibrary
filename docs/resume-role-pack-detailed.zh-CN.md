## 1. 前端岗位详版（Frontend Engineer）

## 1.1 项目定位（前端视角）
- 这是一个 AI Skill 的可视化 IDE 产品，核心前端价值是把“技能创建 -> 导入 -> 审查 -> 安装 -> 模拟”做成一条连贯、可交互、可解释的用户流程。
- 你在前端岗里要强调：复杂状态管理、交互流程设计、浏览器能力调用、可视化风险反馈、组件与页面架构能力。

### 1.2 简历长版（可直接粘贴）
- 负责 Agent Skill Forge 前端架构设计与核心功能开发，基于 React + TypeScript + Vite 实现 Skill Library、Editor、Simulator、Settings 四大模块。  
- 使用 Zustand 统一管理技能数据、主题状态、MCP 连接状态和 UI 通知，构建可维护的客户端状态模型。  
- 实现 Discover 多源聚合能力（本地索引、GitHub Repo、第三方市场源），并对技能数据进行标准化和去重处理。  
- 设计并落地技能安全评分可视化机制：展示 score/risk level/hard-trigger，并在安装流程中执行高危拦截策略。  
- 打通浏览器 File System Access API，实现本地目录授权、技能文件读写、目录树遍历和多文件编辑体验。  
- 完成跨工具安装交互（20 个目标），支持一键写入多个主流 Agent/IDE 技能目录。  
- 设计并实现三套主题系统与 p5 动态背景渲染机制，提升产品视觉辨识度与沉浸式交互体验。  
- 推进工程规范：TypeScript 严格模式、ESLint、模块化组件分层，提升迭代效率与可维护性。  

### 1.3 简历中版（推荐）
- 主导 AI Skill IDE 前端实现，完成技能创建、导入、审查、安装全流程页面与交互。  
- 基于 Zustand 构建状态管理体系，并打通 File System Access API 实现本地目录双向同步。  
- 实现技能安全评分与硬阻断安装策略，提升技能安装前的风险可视化与可控性。  
- 完成多源 Discover 聚合与去重，以及跨工具安装目标映射与可视化配置。  

### 1.4 招聘平台短版（3 条）
- 负责 AI Skill IDE 前端架构和核心页面开发，覆盖创建、导入、编辑、安装全流程。  
- 使用 React + TypeScript + Zustand 实现复杂状态管理和安全评分可视化。  
- 基于 File System Access API 完成本地目录同步和多文件编辑体验。  

### 1.5 面试 90 秒讲述（前端岗）
- 我做了一个面向 AI Agent Skills 的可视化 IDE，重点在前端端到端体验。  
- 在架构上我用 React + TypeScript + Zustand 组织四个核心页面，解决了技能从发现到安装的完整交互链路。  
- 在能力上我打通了 File System Access API，让浏览器可以在授权前提下直接读写本地技能目录，这样用户不用频繁上传下载文件。  
- 在安全上我把规则扫描结果做成可视化评分，并在 UI 侧对高危技能做硬阻断，保证“先审查再安装”。  
- 在体验上实现了多主题和动态背景，让产品不像普通表单工具，而是可持续迭代的 IDE 型前端应用。  

### 1.6 面试高频追问（前端岗）

#### Q1：你怎么拆前端模块？
- 回答要点：
  - 页面层：Library / Editor / Simulator / Settings
  - 状态层：skillStore + mcpStore
  - 能力层：fsCore / securityScan / skillTargets
  - 组件层：通用组件 + IDE 子组件
- 加分表达：
  - “我把易变逻辑下沉到 lib 和 store，页面尽量只管编排和交互。”

#### Q2：为什么选 Zustand？
- 回答要点：
  - 状态规模中等，Zustand 足够且开发效率更高
  - `persist` 满足本地持久化
  - 减少 Redux 模板代码，迭代快

#### Q3：如何保证前端可维护性？
- 回答要点：
  - 类型约束 + 统一 Skill 数据结构
  - 页面与能力模块分离
  - 关键路径（导入/安装/保存）有明确错误提示与回退

### 1.7 前端岗关键词池
- React, TypeScript, Vite, TailwindCSS, Zustand, State Management, Component Architecture, Browser APIs, SPA, Interaction Design, UI Engineering

## 2. 全栈岗位详版（Full-Stack Engineer）

## 2.1 项目定位（全栈视角）
- 这是一个“前端 IDE + 本地代理服务 + 协议接入”的端到端项目。
- 全栈岗重点强调：前后端联动、协议桥接、数据规范化、风控策略、交付闭环。

### 2.2 简历长版（可直接粘贴）
- 负责 Agent Skill Forge 的全栈设计与实现，搭建前端 IDE（React + TS）与本地代理服务（Node + Express）协同架构。  
- 在浏览器端打通 File System Access API，完成目录授权、技能扫描解析、落盘写回和多文件目录管理。  
- 设计并实现 MCP 接入链路：通过本地代理完成 SSE <-> stdio 桥接，支持前端动态连接 MCP 服务并拉取工具列表。  
- 构建多源导入流程（本地索引、GitHub、外部市场源），统一映射为内部 Skill 数据结构，完成字段标准化与去重策略。  
- 建立规则化安全扫描模块（11 条规则），输出风险评分、风险类别与硬触发标识，并在安装前执行策略拦截。  
- 抽象跨工具安装目标体系（20 个 target），支持“sync-major”批量分发到多个 Agent/IDE 技能目录。  
- 处理端到端异常与容错：目录授权失败、导入失败、网络源不可用、安装失败等场景的降级与提示机制。  
- 基于 TypeScript/ESLint 推进工程化约束，确保多模块协作下的代码一致性与可维护性。  

### 2.3 简历中版（推荐）
- 端到端实现 AI Skill 平台：前端 IDE + Node 本地代理 + MCP 协议接入。  
- 打通浏览器本地目录读写能力，实现技能扫描、编辑、回写和多文件结构管理。  
- 建立技能导入标准化与安全门禁体系（规则扫描 + 硬阻断 + 跨工具安装映射）。  
- 通过 SSE <-> stdio 桥接实现 MCP 工具链联调，并提供可视化配置与连接状态反馈。  

### 2.4 招聘平台短版（3 条）
- 完成 AI Skill 平台全栈落地：React 前端 + Node/Express 本地代理。  
- 实现 MCP 协议桥接（SSE <-> stdio）与多源技能导入标准化。  
- 建立规则扫描与安装阻断机制，并支持 20 个安装目标跨工具分发。  

### 2.5 面试 90 秒讲述（全栈岗）
- 这个项目我做的是端到端闭环：前端 IDE 负责可视化编排，后端本地代理负责协议接入和能力桥接。  
- 浏览器侧通过 File System Access API 在用户授权后读写本地技能目录，解决了传统上传下载带来的流程割裂。  
- 服务侧用 Node + Express 搭了本地代理，使用 SSE 对浏览器提供连接，再通过 stdio 对接 MCP 进程。  
- 数据侧我做了统一 Skill 模型，支持多源导入并去重；安全侧做了规则扫描和硬阻断策略，确保高危技能不能直接安装。  
- 最终形成“编辑-审查-安装-联调”闭环，既能演示也能实际使用。  

### 2.6 面试高频追问（全栈岗）

#### Q1：为什么要做本地代理？
- 回答要点：
  - 浏览器不能直接管理本地进程
  - 协议桥接和跨域请求需要中间层
  - 本地代理可统一封装连接逻辑与错误处理

#### Q2：端到端最难的地方是什么？
- 回答要点：
  - 多源数据标准化（字段不统一）
  - 协议连接稳定性（连接生命周期管理）
  - 安全和可用性的平衡（阻断策略 + 用户体验）

#### Q3：如果生产化，先做什么？
- 回答要点：
  - 代理鉴权、命令白名单、域名白名单
  - CI 质量门禁（lint/build/test）
  - 关键链路可观测性（日志与错误上报）

### 2.7 全栈岗关键词池
- Full-Stack, React, TypeScript, Node.js, Express, SSE, MCP, Protocol Bridge, Data Normalization, Security Gate, End-to-End Delivery

### 3.6 面试高频追问（AI 工程岗）

#### Q1：为什么规则引擎优先于 LLM 审核？
- 回答要点：
  - 第一层风控必须稳定、可解释、低成本
  - 规则引擎适合做硬约束
  - LLM 更适合第二层语义审查与解释增强

#### Q2：如何定义“可治理”的 Skill？
- 回答要点：
  - 有结构化元数据（name/allowed-tools/context/agent）
  - 有可执行边界（工具权限/执行上下文）
  - 有可审查策略（规则扫描/阻断/来源可追踪）

#### Q3：这个项目在 AI 体系里的定位？
- 回答要点：
  - 位于模型调用上层的“能力编排与治理层”
  - 连接技能资产、工具协议和运行策略
  - 降低 Agent 能力扩展和迁移成本

## 4. 三套共用的专业名词补充（面试可背）

### 4.1 Skill Frontmatter
- 定义：技能文档头部的结构化配置，用于描述技能元信息和执行约束。
- 价值：把“提示词”升级为“可治理资产”。

### 4.2 Hard-Trigger
- 定义：命中高危规则后直接阻断，不进入后续安装流程。
- 价值：把安全前移到安装前阶段。

### 4.3 Risk Score
- 定义：基于规则权重计算的风险分数（0-100），辅助分级决策。
- 价值：让风控策略可视化、可解释、可沟通。

### 4.4 MCP Bridge
- 定义：通过本地代理把浏览器侧连接桥接到 MCP 进程通信通道。
- 价值：实现工具链可接入与可视化联调。

### 4.5 Multi-Source Ingestion
- 定义：从多个来源导入技能并统一标准化处理。
- 价值：保证技能供给持续扩充且结构一致。

## 5. 使用建议（怎么裁剪到一页简历）

1. 只保留一个岗位版本在主简历，避免信息冲突。  
2. 长版放作品集/附件，中版放简历正文，短版放招聘平台项目框。  
3. 每条经历保持“动作 + 技术 + 结果”结构。  
4. 指标只用可验证数据（如 11 条规则、20 个目标、889 条聚合技能）。  
5. 面试时优先讲“设计取舍”和“风险控制”，不要只讲功能堆叠。
